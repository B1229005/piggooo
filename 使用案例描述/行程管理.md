# 📅 行程管理功能：使用案例描述 (Use Case Specification)

| 欄位名稱 | 最完整、詳細之內容說明 |
| :--- | :--- |
| **使用案例名稱** | 將景點排入指定天數 (Add to Itinerary) |
| **主要參與者** | 群組成員 |
| **前置條件** | 1. 使用者已成功登入 PIGGO 系統。<br>2. 使用者已進入或建立一個旅遊群組計畫。<br>3. 系統已產生 AI 推薦卡片，或聊天室投票已完成。 |
| **觸發條件** | 使用者主動點擊 AI 推薦卡片的「直接加入行程」按鈕，或系統偵測到聊天室投票活動截止。 |
| **基本流程<br>(正常情節 / Normal Flow)** | **1. 觸發與封裝**：系統偵測到觸發信號後，立即將景點資料（名稱、地圖座標、預估停留時間）封裝為系統標準的 `ItineraryItem` 資料物件。<br>**2. UI 彈窗互動**：前端 Flutter 觸發顯示一個 UI 彈出視窗，顯示此趟旅程的所有天數列表，要求使用者指派旅遊天數（例如：Day 1、Day 2 或 Day N）。<br>**3. 天數指派**：使用者在介面上滑動並選定預計排入的天數，按下「確認加入」按鈕。<br>**4. 網路與寫入**：系統發出非同步請求，將該 `ItineraryItem` 物件推送到 Firebase Cloud Firestore 中的 `Plan` 集合。<br>**5. 狀態通知**：後端成功寫入後，底層的 `PlanProvider` 核心會立刻自動被觸發，呼叫 `notifyListeners()` 全域通知機制。<br>**6. 即時重新渲染**：群組內所有成員手機上的狀態管理模組監聽到變更，自動將新景點加入本地的行程清單陣列，【行程表分頁 UI】在毫秒級內即時重新渲染，長出精緻的景點卡片。 |
| **例外流程<br>(例外情節)** | **例外情節 A：使用者於中途點擊取消動作**<br>3a. 使用者在天數選擇視窗中，因為改變心意而點擊了「取消」或視窗外部。<br>3b. 前端 Flutter 關閉彈出視窗，系統立即中斷排程流程。<br>3c. 介面安全退回原瀏覽頁面，不對後端 Firebase 資料庫進行任何寫入與變更。<br><br>**例外情節 B：寫入時網路連線異常斷線**<br>4a. 系統嘗試將資料推送到雲端時，偵測到實體手機網路斷線。<br>4b. 系統自動攔截錯誤，啟用本地快取防護機制，將資料暫存於手機記憶體。<br>4c. 前端畫面下方彈出 Snackbar 微幅提示：「目前網路連線異常，系統已將行程暫存，恢復連線後將自動完成同步」，確保 App 不會崩潰卡死。 |
| **成功保證** | 雲端與本地的群組計畫資料庫成功同步，群組內所有成員的行程表皆呈現最新的排程狀態。 |
