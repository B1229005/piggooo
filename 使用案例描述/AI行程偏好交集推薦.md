# 🧠 AI 行程偏好交集推薦：使用案例描述 (Use Case Specification)

| 欄位名稱 | 詳細之內容說明 |
| :--- | :--- |
| **使用案例名稱** | 發起 AI 群組偏好分析 (AI Preference Recommendation) |
| **主要參與者** | 群組成員、PIGGO 系統核心（AI 引擎） |
| **前置條件** | 群組內的所有成員皆已在【開始使用】的 Onboarding 階段完成系統內建的「旅行人格測驗」，且 Firebase 資料庫中已為每個人存入專屬的偏好標籤（如：小資省錢、網美打卡等）。 |
| **觸發條件** | 使用者在群組計畫編輯頁面中，主動點擊了「✨ AI 群組偏好分析」功能按鈕。 |
| **基本流程<br>(正常情節)** | **1. 發出運算請求**：使用者點擊按鈕，系統向後端 AI 模組發出非同步運算請求。<br>**2. 遍歷並撈取標籤**：系統自動遍歷當前群組內的所有成員 ID，從資料庫中完整抓取每個人當時做旅行人格測驗所產生的人格特質標籤。<br>**3. 交叉矩陣運算**：AI 推薦引擎將 these 多元的標籤建立交叉比對矩陣，運用自然語言處理（NLP）對跨平台抓取的非結構化旅遊資料進行權衡，算出一組能滿足所有人偏好的「最大公約數景點」（例如：選出免門票、好拍照又美食多的台南神農街）。<br>**4. 營業時間 API 比對**：AI 自動串接外部地圖與店家資訊 API，即時讀取該景點當天的最新營業時間，確認其正常營業。<br>**5. 卡片渲染與理由陳列**：前端 Flutter 接收到 JSON 資料，渲染出精緻的 AI 推薦卡片，並在畫面上用文字清晰列出符合群組裡每一位成員人格測驗結果的客觀理由，最後提供「直接加入行程」與「加入投票選項」的雙重決策按鈕。 |
| **例外流程<br>(例外情節)** | **例外情節 A：群組成員標籤極端衝突（零交集）**<br>3a. AI 引擎在比對標籤時，發現群組成員偏好完全相反（例如：有人完全不花錢、有人只去高級餐廳），導致矩陣運算不出合理的交集。<br>3b. 系統自動啟動演算法防護機制，自動切換為「熱門通用防禦方案」。<br>3c. 改為篩選出該旅遊地區在各大社群綜合評價最高、容錯率最大的 3 個經典地標（如安平老街），並在卡片提示：「偵測到群組偏好較多元，AI 為您推薦當地最熱門的必去行程」，避免系統卡死。<br><br>**例外情節 B：偵測到目標店家臨時公休**<br>4a. AI 在串接 API 檢查時，發現預計推薦的景點當天因內部整修、公休或歇業而未開放。<br>4b. 系統在背景自動將該地點從推薦清單中剔除。<br>4c. 流程自動導回基本流程第 3 步，由 AI 模組重新進行交集計算，確保最終呈現在使用者手機上的卡片絕對是當天去得了的有效景點。 |
| **成功保證** | 成功將極具參考價值的 AI 個人化分析結果呈現在畫面上，並提供流暢的資料結構以便隨時拋轉至行程表或聊天室投票。 |
