# 👥 群組討論與投票功能：使用案例描述 (Use Case Specification)

| 欄位名稱 | 詳細之內容說明 |
| :--- | :--- |
| **使用案例名稱** | 參與聊天室景點投票 |
| **主要參與者** | 群組成員 |
| **前置條件** | 1. 特定群組的內建聊天室中，已存在一個由成員發起、正在進行中的景點投票活動。<br>2. 該投票活動尚未超過設定的截止時間。 |
| **觸發條件** | 群組成員在聊天室內瀏覽到投票組件，並用手指點擊其中一個景點選項按鈕。 |
| **基本流程<br>(正常情節)** | **1. 擷取點擊**：成員在聊天室看見投票卡片，點選心儀的景點選項。<br>**2. 時效驗證**：系統首先動態檢查當前伺服器時間，確認該投票活動仍處於有效時間內。<br>**3. 身分與紀錄檢查**：系統撈取該成員的唯一使用者識別碼（UID），向資料庫比對該投票活動的已投票清單，確認其尚未有投票紀錄。<br>**4. 原子操作計票**：系統將成員 UID 寫入已投清單，並呼叫 Firebase 進行資料庫原子操作（FieldValue.increment），讓該選項票數立即 +1，確保多人同時投票時數據不打架。<br>**5. 即時流監聽**：所有群組成員手機前端的 `StreamBuilder` 機制即時捕捉到 Firestore 的動態變更。<br>**6. 動態排版刷新**：`VoteProvider` 更新本地狀態，系統自動計算出最新百分比，群組全員畫面的【投票進度條】產生流暢的動畫位移，即時顯示最新票數。 |
| **例外流程<br>(例外情節)** | **例外情節 A：系統攔截重複投票**<br>3a. 系統檢查發現該 UID 帳號在該投票紀錄中已經存在。<br>3b. 系統立即攔截並阻斷本次投票動作，拒絕更新 Firebase 的計數器。<br>3c. 畫面上跳出提示訊息：「您已進行過投票，不可重複投票」，元件維持原狀，確保一人一票的數據公正性。<br><br>**例外情節 B：使用者嘗試對逾期活動投票**<br>2a. 成員點擊選項時，系統檢查發現當前時間已超過發起人設定的截止時間。<br>2b. 系統拒絕處理該點擊，並自動將該聊天室投票組件的狀態變更為「已截止（Disabled）」。<br>2c. 畫面呈現灰色凍結狀態，並彈出提示：「此投票活動已結束，請檢視最終開票結果」，引導系統走向行程排定流程。 |
| **成功保證 (Post-conditions)** | 投票票數獲得安全且正確的累積，群組全員的介面皆能無延遲地看到絕對公平的表決結果。 |
